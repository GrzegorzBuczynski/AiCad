Project Overview

OVERVIEW

This document is a structured guide for prompting an LLM (Claude, GPT-4, etc.) to build a production-grade parametric CAD application step by step. Each phase produces a running deliverable that extends the previous one. The guide is intentionally ordered from generic to specific — the further you read, the more detailed the prompts become.

INFO
This is not executable code. It is a prompt engineering guide — each section tells you exactly how to frame your LLM prompts so the model builds each subsystem correctly, in order, with proper context handoff between phases. Local Git repository workflow is mandatory for AI-proposed changes.

Core Design Goals

UI Mouse + Keyboard Driven

Primary interaction mode mirrors CATIA V5: toolbar commands, 3-button mouse navigation, hotkeys for sketch and feature operations.

JS JSON Model Format

Every model is serialized to/from a human-readable JSON document. Users can edit models with a text editor; AI proposes JSON changes as Git commits, never as silent in-place edits.

AI Integrated LLM Panel

Built-in chat panel connects to any OpenAI-compatible server by IP. The AI generates change proposals in pull-request style, checker validates them, and user decides whether to merge.

VK Vulkan / DirectX Graphics

Default renderer is Vulkan 1.3. A compile-time option switches to DirectX 12 (D3D12). Abstracted via a thin RHI (Render Hardware Interface) layer.

3D CATIA V5 UX (MVP Scope)

Layout follows CATIA V5: feature tree on left, 3D viewport center, specification panel right. SpaceMouse integration is explicitly deferred until after MVP.

MOD 2-Module Fault Isolation

Main module orchestrates UI and task dispatch, worker module computes geometry/solver steps. If worker crashes, main process restarts it and retries with the same parameters.

PP Performance First

C++20 throughout. SIMD for geometry math. Async geometry rebuild on worker threads. GPU culling via compute shaders. Target: 60 FPS at 1M+ faces.

System Architecture

ARCH

The system is divided into five vertical layers and two runtime modules. Module 1 (main orchestrator) owns UI/state/undo, module 2 (geometry worker) computes solver and topology. If module 2 fails, module 1 restarts it and resubmits the queued request with bounded retry policy.

03

Model File Format (JSON)

SPEC

The model file is the universal interface between human editing, AI proposals, and the CAD kernel. The schema is hierarchical (not flat): downstream features reference upstream feature IDs (for example, Extrude references a Profile/Sketch ID). AI never applies changes directly to active model state; it submits repository changes for review and merge.

TIP
Keep the JSON schema simple and stable. Assign a schema_version field from day one so the LLM can reason about backward compatibility when you upgrade the format.
// example: bracket_v1.cadmodel.json { "schema_version": "1.0", "name": "Mounting Bracket", "units": "mm", "parameters": { "base_width": 60.0, "wall_thickness": 3.5, "height": 45.0 }, "features": [ { "id": "feat_001", "type": "Sketch", "plane": "XY", "entities": [ { "type": "Rectangle", "x":0, "y":0, "w":"#base_width", "h":50 } ] }, { "id": "feat_002", "type": "Extrude", "profile_id": "feat_001", "depth": "#height", "direction": "normal" }, { "id": "feat_003", "type": "Fillet", "edges": "all_vertical", "radius": "#wall_thickness" } ], "metadata": { "author": "user", "modified": "2025-04-18T12:00:00Z", "source": "llm_assisted" } }

Parameter references use the #param_name prefix so the kernel can distinguish literal values from references. Feature references use explicit IDs (for example profile_id) so dependencies remain deterministic. Rebuild delegation is intent-driven: rebuild happens when the user or orchestrator explicitly asks, not on every detected change.

P1

Phase 1 — Foundation

WEEKS 1–4
P1

Goal: A renderable, interactive window

By end of Phase 1 you have: a Vulkan (or DX12) window rendering a 3D grid, an ImGui sidebar, mouse orbit/pan/zoom camera, and basic keyboard shortcut dispatch. No geometry yet — just the shell the rest will live in.

4 WEEKS

1.1 — Window & Application Shell

The window system must be initialized before any renderer code. Prompt the LLM to create this before anything Vulkan-specific to avoid dependency tangles.

  • SDL3 window with Vulkan surface capability and a hidden DX12 fallback path behind #ifdef USE_DX12
  • Main loop: poll events → update → render → swap, targeting 60 Hz with frame-time budget tracking
  • Windowed and fullscreen mode toggle via F11
  • Config file (settings.json) for renderer choice, window size, and LLM server IP/port

1.2 — Vulkan Render Pipeline

Vulkan setup is verbose. Give the LLM the exact feature list so it doesn't hallucinate extension names. Use Dynamic Rendering (no render pass objects) to keep the code manageable.

  • Instance → Physical device selection (prefer discrete GPU) → Logical device with graphics + transfer + compute queues
  • Swapchain with mailbox or FIFO present mode (user-selectable in settings)
  • VkDynamicRendering extension — no VkRenderPass objects; attach color/depth at draw time
  • Mesh pipeline: vertex buffer (float3 pos, float3 normal, float2 uv), index buffer, push constants for MVP
  • Wireframe overlay: separate pipeline with polygonMode = VK_POLYGON_MODE_LINE
  • ImGui Vulkan backend integrated as a secondary pass after geometry
WARN
Always ask the LLM to add Vulkan validation layer support (VK_LAYER_KHRONOS_validation) gated by a DEBUG compile flag. Validation catches driver-undefined behavior that would otherwise cause silent corruption or crashes.

1.3 — Input & Camera System

The camera must feel like CATIA V5: middle-mouse orbit, middle+Shift pan, scroll zoom. Document the exact mouse bindings in a comment block in the code so the LLM remembers them in future sessions.

  • ArcBall orbit: middle mouse drag rotates around target point
  • Pan: middle mouse + Shift translates target
  • Zoom: scroll wheel or right-mouse drag (CATIA style, not FPS zoom)
  • Fit-to-all: F key frames all visible geometry
  • View presets: numpad 1/2/3 = front/top/right, numpad 0 = isometric

P2

Goal: Sketch → Extrude → Feature tree → JSON roundtrip

Phase 2 builds the parametric CAD kernel. By the end you can create a sketch, apply constraints, extrude it into a solid, see the feature tree update, save to JSON, reload, and have the model regenerate correctly from scratch.

2.1 — Geometry Engine (OCCT or Custom BRep)

Open CASCADE Technology (OCCT) is the industrial-strength choice. If the LLM struggles with OCCT API verbosity, you can scaffold a simpler custom BRep for Phase 2 and migrate to OCCT in Phase 3. Either way, keep geometry behind an interface:

Define the geometry abstraction layer. File: src/geometry/IGeometryKernel.hpp Interface methods (pure virtual): createBox(float w, float h, float d) → SolidHandle createExtrude(const Profile&, float d) → SolidHandle createRevolve(const Profile&, float ang)→ SolidHandle booleanUnion(SolidHandle a, b) → SolidHandle booleanCut(SolidHandle base, tool) → SolidHandle tessellate(SolidHandle, float chord) → MeshData computeAABB(SolidHandle) → AABB Implementations: - OcctKernel.hpp/.cpp using Open CASCADE 7.7 - FallbackKernel.hpp/.cpp using simple analytic primitives only // SolidHandle is an opaque uint32_t ID — kernel manages lifetime // MeshData: std::vector<Vertex> + std::vector<uint32_t> indices // Tessellation runs async on thread pool, callback with MeshData when done

2.2 — Sketch Subsystem

The sketch editor is the most complex UI component. It needs a 2D constraint solver, entity drawing in the 3D viewport projected onto a plane, and a properties panel for dimensions.

1

Sketch Entity Model

Lines, arcs, circles, points, construction geometry. Each entity has an ID and a set of free parameters (endpoints, radius, angle).

2

Constraint Solver

Geometric constraints: coincident, parallel, perpendicular, tangent, equal length, fixed. Dimensional: distance, angle, radius. Uses Newton-Raphson or SQP solver.

3

Sketch-to-Profile

Extract closed wire profiles from solved sketch entities. Handle multiple loops (for holes). Output: Profile struct for the geometry kernel.

4

Sketch UI Layer

Overlay drawn with ImGui DrawList on the viewport when sketch mode is active. Snap to grid, snap to entity endpoints, dimension annotation rendering.

Implement the 2D sketch subsystem. Files: src/sketch/ — SketchEntity, SketchConstraint, SketchSolver, SketchDocument, SketchView SketchEntity variants (use std::variant or virtual): PointEntity { vec2 pos } LineEntity { vec2 p1, p2 } ArcEntity { vec2 center; float r, angle_start, angle_end } CircleEntity { vec2 center; float radius } SketchConstraint variants: Coincident, Parallel, Perpendicular, Tangent, EqualLength, FixedPoint, HorizontalConstraint, VerticalConstraint, DistanceDim { float value }, AngleDim { float degrees }, RadiusDim { float value } Solver: Use incremental Newton-Raphson. DOF counter: display in status bar (0 = fully constrained, green; >0 = blue; -N = over-constrained, red). UI (SketchView): Draw via ImDrawList on viewport overlay. Colors: unconstrained = blue, constrained = black/white, selected = orange, construction = dashed gray. Grid: 1mm minor, 10mm major. Snap within 5px screen distance. // Sketch is active while a Sketch feature node is double-clicked in FeatureTree // ESC closes sketch, triggers rebuild of dependent features

2.3 — Feature Tree & Parametric Rebuild

Implement the parametric feature tree and rebuild pipeline. Files: src/model/FeatureTree.hpp/.cpp, FeatureNode.hpp Feature types (Phase 2 scope): SketchFeature, ExtrudeFeature, RevolveFeature, FilletFeature, ChamferFeature, ShellFeature, HoleFeature, MirrorFeature FeatureNode base: - id: uint32_t (auto-incremented) - name: std::string (user-editable) - state: enum { Valid, Warning, Error, Suppressed } - children: std::vector<FeatureNode*> Rebuild pipeline: Rebuild delegation is intent-driven (user action or explicit orchestrator command). Support full rebuild and partial rebuild from any selected feature/node downward. Rebuild executes top-down in topological order on a thread pool. On completion: tessellate() → upload new mesh to GPU → repaint. Fault isolation: All solve/build steps execute in GeometryWorker (module 2) via IPC. If worker dies, MainOrchestrator (module 1) relaunches worker and retries once with same parameters. If retry fails, feature state = Error and UI remains responsive. FeatureTree Panel (ImGui): Tree view with icons per feature type. Right-click context menu: Edit, Rename, Suppress/Unsuppress, Delete, Reorder (drag). Double-click Sketch → enter sketch mode. // Undo/redo: command pattern, store JSON snapshot per step (max 50 steps)

Phase 3 — AI/LLM Integration & Polish

WEEKS 13–20
P3

Goal: LLM can read, describe, and propose model changes via Git workflow

Phase 3 wires the LLM server to the model JSON, adds the AI chat panel, implements a PR-style review gate for AI changes, and polishes CATIA-style UX for MVP (without SpaceMouse integration).

8 WEEKS

3.1 — LLM Bridge Module

The LLM bridge is an async HTTP client that talks to any OpenAI-compatible endpoint. It serializes model context, sends the user request, and converts the AI response into a Git change proposal that must pass checker validation and explicit user approval.

Implement the LLM bridge module. Files: src/ai/LlmBridge.hpp/.cpp, src/ai/LlmConfig.hpp, src/git/LocalRepoManager.hpp/.cpp, src/ai/ChangeProposal.hpp LlmConfig: base_url: std::string (e.g., "http://192.168.1.10:11434") ← user-set in Settings api_key: std::string (empty = no auth, for local servers) model: std::string (e.g., "gpt-4o" or "llama3.3") max_tokens: int (default 4096) temperature: float (default 0.2 for deterministic CAD edits) LlmBridge::sendMessage(user_text, model_json_snapshot) → std::future<LlmResponse> System prompt injected automatically: "You are a CAD assistant. The current model is provided as JSON below. To modify the model, output a unified diff and commit message proposal. Do not assume direct write access to production model files. Include title, rationale, risk notes, and test checklist in PR-style format. For questions about the model, answer in plain text." Response parsing: Parse proposed diff + commit metadata from assistant reply. Create local feature branch (e.g., ai/proposal-YYYYMMDD-HHMM). Commit AI proposal in local repo. Run checker pipeline (schema validator + dependency validator + optional build/tests). Open PR-style preview panel with changed files, checks, and risk summary. Only on user Accept/Merge: fast-forward or squash-merge into active branch, then trigger reload/rebuild. On Reject: keep branch for audit or delete branch per user choice. // HTTP: use cpp-httplib (header-only). Streaming responses: update chat panel per chunk. // Thread safety: model JSON snapshot taken under read-lock before sending
TIP
When prompting the LLM to build this module, emphasize the streaming response requirement. The user should see the AI reply character-by-character like a normal chat — not wait for the full response before it appears. Use the stream: true SSE endpoint.

3.3 — JSON Edit → Live Regen

The third interaction mode: the user edits the .cadmodel.json file directly in any text editor. AI edits still flow through the local Git proposal pipeline. The file watcher detects external changes and queues intent-driven rebuild suggestions. Bad JSON must never crash the app.

Implement the JSON file watcher and hot-reload pipeline. File: src/io/FileWatcher.hpp/.cpp Mechanism: Poll file mtime + size every 500ms on background thread. Debounce: wait 200ms after last change before triggering reload (handles editors that write in two steps). On change: read file → parse JSON → validate schema → if valid, post ReloadModelEvent to main thread queue. Main thread on ReloadModelEvent: 1. Store current undo snapshot 2. Run deserializer (returns std::expected) 3. If error: show notification toast (red, 5s), keep current model 4. If success: diff old vs new feature list, prepare rebuild suggestion (full or branch) 5. Execute rebuild only after explicit user/orchestrator intent confirmation AI-originated changes: Always come from merged local proposal branch. File watcher should tag such changes with commit hash and proposal ID for audit trail. JSON editor panel (optional, in-app): Embedded text editor (ImGui + basic syntax highlight or use ImGuiColorTextEdit library). Built-in schema validator + dependency validator (feature IDs, reference integrity). Cursor-aware proposal toolbox suggests legal fragments for current JSON path. Example: inside Extrude node, proposals include profile_id, direction, draft, operation mode. Example: inside profile/entity lists, proposals include line, sketch, extract snippets. "Apply" button validates first, then requests intent-driven reload/rebuild. Error annotations: red underline on lines that fail schema validation, tooltip shows error. // The file watcher must survive the model file being deleted and recreated (some editors do this) // Use inotify on Linux, ReadDirectoryChangesW on Windows for OS-native events as enhancement

A3

Prompting Best Practices

META

Context Management

LLM context windows are finite. At ~15k lines of C++, the full codebase won't fit. Use these strategies:

  • Always start a new session with the System Prompt for that phase. It re-establishes assumptions without needing to paste code.
  • Paste only the relevant file header (.hpp) when asking to implement a .cpp — the interface is the contract.
  • After a major milestone, ask the LLM to write a 1-paragraph "session summary" you can paste into the next session as a status update.
  • Use the phrase "diff-style only" to prevent the LLM from reprinting entire files when making small changes.

Avoiding Hallucinations

  • Specify exact library versions (e.g., "Open CASCADE 7.7", "ImGui 1.91") — vague version specs lead to hallucinated API names.
  • For Vulkan: paste the exact extension list you want enabled. Never ask "use the best extensions" — the model will invent some.
  • After each code generation, ask: "List every third-party function you called that you aren't 100% sure exists." Then verify those.
  • Run the code before the next session. Paste compile errors back as-is — the LLM fixes them faster with exact error text.

Effective Decomposition

1

Interface First

Always ask for the .hpp (interface) before the .cpp (implementation). Review and correct the interface before writing a line of implementation code.

2

One Subsystem Per Session

Don't ask for "the entire sketch system." Ask for "SketchEntity types and SketchDocument," then a new session for "SketchSolver," then one for "SketchView."

3

Test Before Extending

Write and run the Catch2 tests after each subsystem. Prompt: "Write Catch2 tests for ModelSerializer roundtrip" before moving to the next task.

4

Error-First Thinking

Append "Also describe how this component handles every possible failure mode" to any implementation request. Forces the model to think about error paths it would otherwise skip.

WARN
Large Vulkan init blocks (1000+ lines) should be split into at least 3 prompts: (1) instance + surface + device, (2) swapchain + dynamic rendering setup, (3) pipeline + mesh rendering. Doing it in one prompt reliably produces subtle initialization-order bugs.
